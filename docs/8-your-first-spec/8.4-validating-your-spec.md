# Validating your spec

Validation checks that a spec conforms to the Catenator standard. A valid spec is not necessarily a complete or well-written spec — validation confirms structure and syntax, not quality. A spec can be valid and still have gaps in completeness, precision, or unambiguity. Validation is the floor, not the ceiling.

---

## What validation checks

**Schema conformance** — the YAML structure matches the Catenator JSON Schema. Required fields are present. Field types are correct. Descriptor names are from the defined set or declared as domain descriptors. Meta-vocabulary terms are used as defined.

**Reference resolution** — every `$ref` points to a term that exists in the declared vocabulary. No dangling references.

**Provenance completeness** — `subject`, `author`, `version`, and `date` are present. `version` is a positive integer. `date` is a valid ISO 8601 date.

**Vocabulary declaration** — if `$ref` is used anywhere in the spec, a vocabulary is declared in the provenance block.

---

## Validating with the JSON Schema

The Catenator JSON Schema is in `schema/catenator.schema.json` in the standard repository. Any JSON Schema validator can check a spec against it.

Using the `ajv` CLI:

```bash
ajv validate -s catenator.schema.json -d your-spec.yaml
```

Using Python:

```python
import yaml
import jsonschema

with open("catenator.schema.json") as f:
    schema = json.load(f)

with open("your-spec.yaml") as f:
    spec = yaml.safe_load(f)

jsonschema.validate(instance=spec, schema=schema)
```

A spec that passes schema validation is a valid Catenator spec.

---

## Beyond schema validation

Schema validation is necessary but not sufficient for a buildable spec. After a spec passes validation, the questions that remain are qualitative:

- Is this spec complete enough that an agent can begin?
- Are the terms used precisely enough to implement?
- Are the failure modes named for every descriptor that carries risk?
- Are the views applied where the system's complexity requires them?

These questions are the domain of the -ilities. The intelligence layer measures them. Schema validation cannot answer them — they require either human review or agent-based evaluation against the spec's declared vocabulary and the system it describes.

**Validate first. Then assess. A spec that fails validation cannot be evaluated. A spec that passes validation has not yet been assessed.**
