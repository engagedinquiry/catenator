# Extending descriptors for a domain

The thirteen descriptors are the core vocabulary of any Catenator spec. They are designed to hold any domain without modification. A pharmacy and a SaaS platform use the same descriptors — what changes is the vocabulary those descriptors draw from, not the descriptors themselves.

However, some domains have structural concerns that the core descriptors do not cover precisely enough. The standard accommodates this through two extension mechanisms: subtypes and domain descriptors.

---

## Subtypes

A subtype specializes an existing descriptor for a specific context without replacing it. Subtypes inherit the parent descriptor's requirements and add their own.

A `rule` descriptor covers all constraints the system enforces. In a heavily regulated domain — healthcare, finance, firearms — the distinction between internal business logic and external regulatory obligation matters enough to name explicitly. A subtype makes that distinction:

```yaml
rule:
  name: prior_authorization_requirement
  subtype: regulatory
  source: CMS
  jurisdiction: federal
  effective_date: 2024-01-01
```

Versus:

```yaml
rule:
  name: manuscript_approval_gate
  subtype: internal
  governs: proof
  enforced_by: editor
```

Both are `rule` descriptors. The subtype carries additional required attributes specific to its category.

Subtypes MUST be declared in the domain vocabulary before they can be used in a spec. A subtype not declared in the vocabulary is not valid.

---

## Domain descriptors

A domain descriptor is a new descriptor — not a subtype of an existing one — that a domain requires and the core thirteen do not cover.

Domain descriptors are rare. Before defining one, the question to ask is: can this be expressed through an existing descriptor with a subtype or additional attributes? If yes, use that path. A domain descriptor is only warranted when the concept is structurally distinct — when it has its own attributes, relationships, and constraints that cannot be attached to any existing descriptor without distorting its meaning.

An example of a warranted domain descriptor: `lot` in a pharmaceutical manufacturing domain. A lot is not a model, not a component, not a process. It is a traceable production batch with its own lifecycle, its own regulatory obligations, and its own failure modes. It is structurally distinct. It warrants its own descriptor.

Domain descriptors MUST be:

- Declared in the domain vocabulary with a full definition
- Named in lowercase, following the same convention as core descriptors
- Accompanied by a scope statement explaining why an existing descriptor is insufficient

---

## What cannot be extended

The meta-vocabulary — `spec`, `item`, `subject`, `descriptor`, `attribute`, `value`, `type`, `reference`, `constraint`, `contract`, `relationship` — is fixed. It is the language of specs, not the language of systems. Extending it would break composability across specs and domains.

The normative language keywords — `MUST`, `SHOULD`, `MAY` — are fixed.

**Extend the vocabulary of the system. Never extend the vocabulary of the spec.**
