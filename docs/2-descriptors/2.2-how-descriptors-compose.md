# How descriptors compose

Descriptors do not stand alone. A system is not a collection of isolated parts — it is a set of relationships between parts. Composition is how those relationships are expressed in a spec.

Composition works at two levels.

*Within a spec, descriptors reference each other.*

- An `operation` references the `actor` that initiates it
- A `rule` governs a `model`
- A `process` connects `events`, `operations`, and `actors` into a sequence
- A `risk` attaches to an `integration`

These connections are expressed through the meta-vocabulary — specifically through `relationship`, `reference`, and `constraint`. They are not implied. They are declared.

*Across specs, a spec can reference another spec.*

A system spec for a pharmacy platform can reference a separate spec for its dispensing workflow, which references a separate spec for a drug interaction check. Each spec is complete at its own level of resolution. Together they describe the full system without any single spec becoming unmanageable.

This is composability. Like Lego — every brick is designed to connect with every other. The stud is standardized. The combinations are unbounded.

Composition across specs also means vocabulary terms are shared. A `prior_authorization` defined in a healthcare domain spec means the same thing in every spec that references it. This is what makes the standard semantic — terms carry meaning across boundaries, not just within a single document.

**A spec that composes well is a spec that can grow without breaking.**
