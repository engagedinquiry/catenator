# Failure

The failure view examines what breaks and who bears it.

Failure is a cross-cutting view. Every descriptor has a failure envelope — a set of conditions under which it does not behave as specified. A `model` can be in an invalid state. A `process` can stall or produce the wrong output. An `integration` can become unavailable. A `rule` can be enforced inconsistently. Isolating failure thinking in the `risk` descriptor is insufficient. The failure view distributes it across the entire spec.

---

## What failure covers

| Name | Description |
|---|---|
|**Failure modes** | the specific ways a descriptor can fail to perform as specified. Not a general statement of risk but a located description: this operation fails when this condition is true. This model is invalid when this constraint is violated. This integration fails when the external system returns this response.|
|**Failure conditions** | what causes the failure. Missing input, invalid state, exceeded capacity, dependency unavailability, rule violation, actor acting outside expected bounds. The condition is as important as the mode — it determines how the failure can be detected and prevented.|
|**Failure consequences** | what the failure produces. A wrong output, a stalled process, an inconsistent state, a violated contract. Who or what is affected. Whether the failure propagates.|
|**Failure ownership** | who bears the consequence. The actor who initiated the failing operation, the system that holds the failing state, the integration that failed to respond, the organization that wrote the rule that proved insufficient. Ownership is not blame — it is the declaration of who is responsible for detection, response, and recovery.|
|**Recovery** | what the system does when failure occurs. Retry, rollback, compensate, alert, degrade gracefully. Recovery behavior is still behavior — it belongs in the failure view because it only exists in response to a failure condition.|


---

## Failure and risk

The `risk` descriptor names failure at the system level — what can go wrong, how likely, and who bears it. The failure view applied to individual descriptors names failure at the component level. Both are necessary. System-level risk without component-level failure analysis produces specs that identify problems without locating them. Component-level failure analysis without system-level risk produces specs that miss the interactions between failures.

---

## Failure in practice

Examined through the failure view, an `integration` descriptor answers:

- What happens when this external system is unavailable?
- What happens when it returns an unexpected response?
- Does this system retry, fail fast, or degrade?
- Who is notified?
- What state is the system left in?

Examined through the failure view, a `rule` descriptor answers:

- What happens when this rule cannot be enforced?
- What happens when it is violated?
- Who detects the violation?
- What is the consequence for the actor who violated it?

**Failure is not the opposite of design. It is part of it. A system whose failure envelope is unnamed has not been fully designed.**
