# Views

A descriptor names what you are examining. A view is the lens you bring to it.

Every descriptor in a Catenator spec can be examined through five views: Interface, Structure, Behavior, Failure, and Time. A view is not a separate document or a separate pass — it is a dimension of the same spec entry, applied deliberately.

The five views are universal. They are not owned by Catenator — they reflect how systems have always been understood, across architecture, engineering, and design. Catenator formalizes them as a first-class axis of the spec so that no view is accidentally omitted.

---

## The five views

| View | Examines |
|---|---|
| `interface` | how the system meets what is outside it |
| `structure` | what the system consists of internally |
| `behavior` | how the system operates |
| `failure` | what breaks and who bears it |
| `time` | how the system is planned and how it evolves |

---

## How views work in a spec

A spec entry occupies a position on the descriptor axis. The view determines which questions that entry is expected to answer.

The same descriptor examined through different views produces different spec content:

- `actor × interface` — who this actor is to the system, what they can reach, what the system assumes about them
- `actor × structure` — how this actor is represented internally — role, permissions, state
- `actor × behavior` — how this actor moves through the system over time
- `actor × failure` — what happens when this actor acts outside expected bounds
- `actor × time` — how this actor's role was defined, how it has changed, where it is going

No spec is required to apply all five views to every descriptor. A minimal spec applies the views the system's complexity requires. A complete spec applies all five where the stakes warrant it.

**A view is a question you choose to ask. The value is in choosing deliberately.**
