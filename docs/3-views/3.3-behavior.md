# Behavior

The behavior view examines how the system operates.

Behavior is the system in motion — what it does when acted upon, what it does on its own, how it responds to state changes, and how it moves through time during a single interaction or workflow. Where structure describes what is there, behavior describes what happens.

---

## What behavior covers

| Name | Description |
|---|---|
|**Response** | how the system reacts when an actor acts or an event occurs. Given this input, this operation runs. Given this event, this process is triggered. Given this state, this rule applies. Behavior is the mapping from stimulus to response.|
|**State transitions** | how the system's objects change state. A `manuscript` moves from `submitted` to `under_review` to `approved`. The states belong to the structure view. The transitions — what triggers them, what conditions must be true, what they produce — belong to behavior.|
|**Execution** | how operations run. What they receive, what they do, what they return. The sequence of steps within a process. The conditions under which a step is skipped or repeated. The logic that is not visible in the structure.|
|**Autonomy** | what the system does without being asked. Scheduled operations, background processes, automated rules that fire on state changes. Autonomous behavior is still behavior — it must be named and specified, not left implicit.|


---

## Behavior and reproducibility

The behavior view is the primary site of reproducibility. A spec that specifies behavior completely enough produces the same system every time it is built from. Vague behavior specifications produce systems that diverge. The test of a behavior specification is whether an agent can implement it without making assumptions.

---

## Behavior in practice

Examined through the behavior view, an `operation` descriptor answers:

- What input does this operation receive?
- What does it do with that input?
- What does it return?
- What state does it leave the system in?
- What rules govern it?
- What events does it trigger?

Examined through the behavior view, a `process` descriptor answers:

- What initiates this process?
- What sequence of steps does it follow?
- What decisions does it make and on what basis?
- What are the exit conditions — both success and failure?

**Behavior is what makes a spec executable. A spec without it is a map without roads.**
