# Interface

The interface view examines how the system meets what is outside it.

Interface is the broadest of the five views. It covers everything that constitutes the meeting between the system and what is not the system — actors, context, situation, dependencies, and the conditions under which interaction occurs.

---

## What interface covers
| Name | Description |
|---|---|
|**Actors** | who or what the system meets. Every actor is part of the system's interface. The system assumes things about its actors — their knowledge, their context, their intent. Those assumptions belong in the interface view.|
|**Context** | the conditions under which the interaction occurs. A system deployed in a hospital emergency department has a different interface than the same system deployed in a GP clinic. The software may be identical. The interface is not.|
|**Situation** | how the system positions itself in the actor's world. Does it belong there or displace something else? Does it recede into the background or demand attention? Situation is the relational property of the system — it only exists in the presence of an actor in a specific context.|
|**Boundaries** | where the system ends and something else begins. What passes through and what does not. What the system is responsible for and what it is not. A boundary is not just a technical edge — it is a declared scope.|
|**Dependencies** | what the system requires from outside itself in order to function. Every dependency is an interface obligation. If the dependency fails, the interface fails.|

---

## Interface and intent

Intent is a property of the interface view. Intent describes what the system communicates to the actor — what it asserts, what it demands, what it withholds, how it positions the actor relative to itself. Intent cannot be specified without naming the actor it is for.

Intent is not designed into the system directly. The designer shapes the conditions. The actor and context produce the actual intent at the moment of encounter. The interface view is where those conditions are named.

---

## Interface in practice

Examined through the interface view, a `service` descriptor answers:

- What does this service expose, and to whom?
- What does it assume the caller knows?
- What contract does it hold with its consumers?
- What context does a caller need to interact with it correctly?

Examined through the interface view, an `integration` descriptor answers:

- What is the boundary between this system and the external system?
- What passes through?
- Who initiates?
- What does each side assume about the other?

**Interface is not the thin technical contract. It is everything that constitutes the meeting.**
