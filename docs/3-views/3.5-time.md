# Time

The time view examines how the system is planned and how it evolves.

Time holds two directions simultaneously. Looking forward, it documents planning — what gets built, when, in what order, and what depends on what. Looking backward, it documents evolution — what the system was, what changed, what drove the change. The same view, both directions.

---

## What time covers

### Forward — planning

| Name | Description |
|---|---|
|**Milestones** | the points in the system's development at which a meaningful state is reached. Not a task list. A milestone is a declaration that something is complete enough to build on. The macro spec is complete. The Editorial domain is specced. The Submission workflow is ready for implementation.|
|**Phases** | the periods between milestones. What is in scope for this phase, what is explicitly out of scope, and what decisions are deferred to a later phase. Phase boundaries are decisions, not just dates.|
|**Dependencies** | the ordering constraints between parts of the system. This component cannot be built until this model is defined. This integration cannot be specced until this service's contract is stable. Planning dependencies are structural — they follow from the spec itself, not from resource availability.|
|**Deferral** | what the spec explicitly leaves unresolved. A deferred decision is not an omission — it is a named gap with a condition for resolution. Deferred until the integration contract is confirmed. Deferred until the regulatory requirement is clarified. Deferral acknowledged is safer than deferral hidden.|


### Backward — evolution

| Name | Description |
|---|---|
|**Versioning** | the history of what the spec was. Every version is immutable. A change to a spec produces a new version. The history of versions is the history of decisions about the system.|
|**Change rationale** | why a spec entry changed. Not just what changed but what drove the change. A rule was tightened because a compliance requirement changed. A model attribute was deprecated because the downstream integration no longer consumed it. The rationale is the context that makes the change legible to anyone who reads the spec later.|
|**Migrations** | what the system must do to move from one version to another. A model attribute that changes type requires a data migration. A rule that changes its enforcement point requires a process change. Migrations are the behavioral consequence of structural evolution.|
|**Deprecations** | what the spec once required and no longer requires. A deprecated descriptor, term, or attribute is not deleted — it is marked and its removal is versioned. Deprecation is the honest acknowledgment that the system has changed.|

---

## Time and GitHub

The time view is where version control and the spec converge. Every commit to a spec file is a timestamped, attributed change to a coordinate. The diff is the delta between two versions. The commit message is the change rationale. The branch is the phase.

A spec structured finely enough — one file per descriptor, one section per view — produces diffs that are meaningful at the coordinate level. A change to `3.1-interface.md` in the `actor` descriptor tells the reader exactly what changed and where. A monolithic spec produces diffs that are difficult to interpret.

---

## Time in practice

Examined through the time view, a `component` descriptor answers:

- When is this component needed — which phase?
- What must be true before it can be built?
- How has its definition changed across versions?
- What drove those changes?
- Is anything about this component currently deferred?

Examined through the time view, a `rule` descriptor answers:

- When was this rule introduced?
- Has its scope or enforcement changed?
- Is it stable or subject to an external dependency that may require revision?

**Time is not a property of the spec. It is a view through which every coordinate can be examined. The framework is invariant. The system moves.**
