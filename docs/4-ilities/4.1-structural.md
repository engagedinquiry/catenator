# Structural -ilities

Structural -ilities are properties the standard must support. A conforming spec — one written according to the Catenator standard — gets these properties by construction. They are not aspirational. They are guaranteed if the spec is valid.

---

## Reproducibility

Given the same spec, the same system can be produced again.

Reproducibility is the foundational guarantee of the Catenator standard. It is what separates a spec from a document. A document is read and interpreted — two readers may produce different systems from the same document. A spec is executed — the same spec produces the same system, regardless of which agent or team builds from it.

Reproducibility requires that the spec be complete enough that no build decision is left to interpretation. Every ambiguity is a reproducibility gap.

---

## Compilability

From any valid spec, build instructions can be generated.

Compilability means the spec is structured so that an agent can derive what to build next without human interpretation. The spec does not list tasks — it describes the system precisely enough that tasks are derivable from it.

A spec that requires a human to translate it into build instructions before an agent can act is not compilable. The translation is the gap.

---

## Traceability

Every part of the system is traceable back to the spec decision that produced it — and every spec is traceable back to its author, version, and domain vocabulary.

Traceability runs in both directions. Forward: given a system component, find the spec coordinate that produced it. Backward: given a spec coordinate, find all system components that were produced from it. A traceable spec makes auditing, debugging, and change management possible at the coordinate level rather than the document level.

---

## Composability

Any spec can connect with any other. Any vocabulary term can be referenced across domains. Systems can be assembled from existing parts without reinventing what already exists.

Composability is the Lego property. Every brick is designed to connect with every other because the stud is standardized. In Catenator, the meta-vocabulary is the stud — it is the connection interface that makes composability possible across specs, domains, tools, and agents.

A spec that composes well references other specs rather than duplicating their content. A vocabulary term defined once is usable everywhere. A component specced in one system can be referenced by another.

---

## Extensibility

New domains, vocabularies, and descriptors can be added without breaking what already exists.

Extensibility means the standard grows without fracturing. A new specimen vocabulary does not invalidate existing specs. A domain descriptor added for a pharmaceutical domain does not affect a spec written for a publishing imprint. The framework accommodates growth by design, not by exception.

---

## Portability

Any conforming agent or tool can read and act on the spec.

Portability means the spec is not locked to a specific tool, platform, or agent. A Catenator spec written today can be read by a tool that does not yet exist. It can be migrated between platforms without loss. It can be passed between agents without reformatting.

Portability is guaranteed by the YAML format and the JSON Schema validation layer. Any system that can parse YAML and validate against the schema can read a Catenator spec.
