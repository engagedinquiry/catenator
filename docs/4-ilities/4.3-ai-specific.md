# AI-specific -ilities

AI-specific -ilities are properties that matter specifically when agents read and act on the spec. They are the bridge between the open standard and the intelligence layer built on top of it.

The structural -ilities guarantee what the standard produces. The qualitative -ilities describe what the author is responsible for. The AI-specific -ilities describe what the intelligence layer measures — the properties an evaluation tool or validation agent looks for when assessing whether a spec is ready to drive a build.


## Interpretability

An agent reading this spec arrives at the same understanding the author intended.

Interpretability is the AI-facing counterpart of unambiguity. Unambiguity is a property the author targets. Interpretability is a property the agent confirms. A spec that is unambiguous to a human reader may still be misinterpreted by an agent if it uses terms that are not in the declared vocabulary, references that are not resolved, or structures that fall outside the schema.

Interpretability is measurable: given a spec, does an agent's summary of its contents match the author's intent? Gaps between the two are interpretability failures.

---

## Configurability

The spec can be used to configure the system's behavior without modifying the system's code.

Configurability describes the degree to which the spec externalizes decisions that would otherwise be hardcoded. A rule declared in the spec rather than embedded in an operation is configurable — it can be updated in the spec without a code change. An actor's permitted operations declared in the spec are configurable. A process's branching conditions declared as rules are configurable.

A highly configurable spec produces a system that is governable through the spec itself. A low-configurability spec produces a system where the spec and the code have diverged — the spec describes what was intended, the code reflects what was built.

---

## Reliability

The spec produces the same system when built by different agents under different conditions.

Reliability is the agent-facing counterpart of reproducibility. Reproducibility is a structural guarantee — the spec is written so that the same system is theoretically producible from it. Reliability is an empirical property — when agents actually build from the spec, do they produce equivalent systems?

Reliability gaps reveal where the spec's structural reproducibility breaks down in practice. A spec that is reproducible in theory but not reliable in practice has precision or unambiguity failures that the structural -ilities did not catch.

---

## Verifiability

The system produced can be tested against the spec that produced it.

Verifiability means the spec is written with enough precision that its claims are testable. A rule that states "a manuscript cannot be approved without a completed proof" is verifiable — a test can check whether the system enforces this constraint. A rule that states "the system handles manuscripts appropriately" is not verifiable — there is no test derivable from it.

Verifiability requires that behavior be stated in terms of observable outcomes, that constraints be stated as conditions that can be checked, and that failure modes be stated specifically enough that a test can reproduce them.

---

## Readiness

The spec is complete and precise enough that an agent can begin building from it without requiring additional input.

Readiness is the composite -ility — it integrates completeness, precision, unambiguity, interpretability, and verifiability into a single assessment. A spec that is ready is one where an agent encounters no unresolved questions, no undefined references, no ambiguous terms, and no missing failure modes before it can produce build instructions.

Readiness is the primary output of the intelligence layer's evaluation. A readiness assessment is not a score — it is a located report: which coordinates are ready, which have gaps, and what would be needed to close them.
