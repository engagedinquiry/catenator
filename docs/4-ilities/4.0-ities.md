# -ilities

The -ilities are quality dimensions. They answer a different question than descriptors and views.

A descriptor names what you are examining. A view names the lens you bring. The -ilities name how well the description holds — how rigorously a coordinate has been specified, and what standard it is held to.

---

## Three groups

Catenator organizes the -ilities into three groups by function.

| Name | Description |
|---|---|
**Structural** — what the standard must support. These are properties the framework guarantees if the spec is written in conformance. They are about the spec as an object — what can be done with it, how it connects to other specs, how it survives being read by different agents and tools.

**Qualitative** — what a well-written spec achieves. These are properties the author is responsible for. The standard cannot enforce them mechanically — they require judgment. A spec can be syntactically valid and still fail on completeness or precision.

**AI-specific** — what the intelligence layer measures. These are properties that matter specifically when agents read and act on the spec. They are the bridge between the open standard and the tooling built on top of it.

---

## How -ilities apply

The -ilities are a third axis of the coordinate space. A spec entry occupies a position on the descriptor axis and the view axis. The -ilities determine the quality standard that entry is held to.

Not every coordinate requires every -ility. A `utility` descriptor examined through the `interface` view needs `completeness` and `unambiguity` — a vague utility statement produces a system that solves the wrong problem. The same entry does not need `compilability` — utility is not directly compiled into build instructions.

Applying -ilities deliberately means knowing which ones matter for a given coordinate and specifying to that standard — not applying all of them uniformly and not ignoring them entirely.

---

## -ilities and practice

The -ilities are established practice. They appear in ISO 25010, TOGAF, SEI quality frameworks, and architecture review processes. Catenator does not invent them. It places them as a first-class axis of the spec so they are applied to located coordinates rather than used as a final-pass checklist on the output document.

The difference: "is this spec complete?" is a document-level question with no actionable answer. "Is this component's behavior description complete?" is a coordinate-level question that produces a specific gap.
