# Qualitative -ilities

Qualitative -ilities are properties a well-written spec achieves. The standard cannot enforce them mechanically — they require judgment from the author. A spec can be syntactically valid and structurally sound and still fail on these properties. They are the author's responsibility.

---

## Completeness

The spec names everything the system requires — nothing essential is absent.

Completeness is not exhaustiveness. A complete spec does not describe every possible state, every edge case, every implementation detail. It describes everything an agent needs to build the system without making assumptions. The test: can an agent encounter an unresolved question while building from this spec? If yes, the spec is incomplete at that coordinate.

Completeness is the most common failure mode of real specs. The missing actor, the unspecified rule, the process with no failure condition — each is a completeness gap that becomes an implementation decision made without authorization.

---

## Consistency

The spec does not contradict itself — across descriptors, across views, across versions.

Consistency failures are often invisible. A rule declares that a manuscript cannot be approved without a completed proof. A process describes approval happening before proof assignment. Both statements are locally coherent. Together they are inconsistent. The system built from them will be wrong in ways that are hard to trace back to the spec.

Consistency requires that every coordinate be read in relation to every other coordinate that touches the same system concern. It is the most cognitively demanding qualitative -ility.

---

## Unambiguity

Every statement in the spec has exactly one valid interpretation.

Ambiguity is the gap between what the author meant and what the agent builds. It lives in vague terms, undefined references, underspecified constraints, and conditions that are true in some contexts and false in others without the context being named.

Unambiguity requires precision in language and explicitness in reference. "The editor reviews the manuscript" is ambiguous — which editor, which review, under what conditions, with what outcome. "The assigned editor performs a completeness review of the submitted manuscript, producing a review record with a status of approved or returned" is unambiguous.

---

## Precision

Every statement in the spec is specific enough to act on — no more, no less.

Precision is distinct from completeness. Completeness is about what is present. Precision is about how well what is present is specified. An attribute declared as `string` is less precise than one declared as `string, max 200 characters, must match pattern [A-Za-z0-9-_]`. Both are present. Only one is precise enough to implement without an additional decision.

Precision also has an upper bound. Over-specifying implementation detail — constraining choices that should belong to the builder — is a precision failure in the other direction. A spec is precisely specified when it closes every decision that belongs to the design and leaves open every decision that belongs to the implementation.

---

## Minimality

The spec contains nothing that is not necessary.

A minimal spec makes every statement count. Redundant content creates consistency obligations — two statements about the same thing must always agree. Superfluous constraints narrow the builder's options without adding value. Explanatory content that belongs in documentation rather than spec adds noise that degrades interpretability.

Minimality does not mean brief. A complex system requires a substantial spec. Minimality means that every line earns its place — that removing it would leave the system underspecified.
