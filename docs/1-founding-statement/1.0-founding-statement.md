# Founding statement — Catenator standard

Software is becoming something you specify, not something you write. The code is still produced — but by agents, tools, and teams working from structured descriptions of what the system must be.

Across the industry, a spec is emerging as the primary artifact of software development — structured, semantic, and machine-readable.

**Catenator is the open standard for writing one.**

---

A **Catenator spec** is the structured description of a system:

- what it consists of
- what it requires
- how its parts relate

expressed in a format that any agent, tool, or team can read and build from.

---

The spec precedes the system. The system is produced from the spec. A Catenator spec must be written so that:

- Given the same spec, the same system can be produced again
- From any valid spec, build instructions can be generated
- The system produced can be tested against the spec that produced it
- Every part of that system is traceable back to the spec decision that produced it — and every spec is traceable back to its author, version, and domain vocabulary

The standard is designed so that:

- Specs compose — like Lego, any spec can connect with any other, any vocabulary term can be referenced across domains, and systems can be assembled from existing parts without reinventing what already exists
- New domains, vocabularies, and descriptors can be added without breaking what already exists
