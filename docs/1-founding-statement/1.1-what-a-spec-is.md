# What a spec is

A spec is a structured description of a system — written for an agent.

It is not instructions. It is not a prompt. It does not tell an agent what to do next. It tells an agent enough about what the system is so that the agent can determine what to do next itself.

A Catenator spec describes a system through some or all of thirteen descriptors: its domain, its models, its components, its services, its operations, its actors, its events, its rules, its processes, its integrations, its utility, and its risks. Each descriptor is a lens. Together they form a picture of what the system consists of, what it requires, and how its parts relate.

A spec is not:

- a PRD — which describes what users need
- a design doc — which describes how something looks or works
- a backlog — which lists what to build next
- a README — which explains what already exists
- a prompt — which instructs an agent in natural language

A spec precedes all of these. It is the structured foundation from which any of them can be generated.

**A Catenator spec is what you give an agent instead of instructions.**
